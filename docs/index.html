AI Security Operating Model

From fragmented controls to assured governance.

AI security capability exists, but it is scattered across the lifecycle and uneven in depth. This model packages 59 capabilities across 8 domains into a single gated workflow that moves an organisation from unknown estate to audit-ready assurance. Identity is treated as the primary perimeter; controls ship as Terraform and policy-as-code; each phase has an exit gate that has to pass before work advances. The gated lifecycle

Four phases run as a continuous loop. Each phase owns two capability domains and ends in an exit gate. Switch to Today's coverage to recolour every domain by current market maturity and read the coverage statement underneath.

Gate D feeds discovery: assurance findings re-open the estate inventory and the loop repeats.
02

Standards each domain answers to

The sell is not the control list, it is defensible mapping. Every domain ties to named standards so audit and regulatory questions have a pre-built answer.

03

How it converts to an engagement

The same model is the sales motion. Sell the diagnostic first, then the target, then the build. Each step has a fixed deliverable a stakeholder can sign off.

04

Built as code, enforced as policy

Controls that live in slideware do not survive contact with a release pipeline. This model assumes every domain is expressed as infrastructure and policy-as-code, with the phase gates wired in as pipeline checks that block a merge or a deploy when criteria fail.

  • Identity (GCP IAM, Entra ID) is the first thing modelled, before network or compute.
  • State files are encrypted, versioned, and access-controlled. Drift detection runs on a schedule.
  • Gates A to D are OPA or Sentinel policies, not review meetings.
  • Secure landing zones carry the baseline so new AI workloads inherit it by default.
